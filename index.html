PARTS - Path-Aware Reliable Transport over SCION

Marten Gartner

2021

Why PARTS?

  • Missing reliable transport protocol for SCION
  • QUIC works well, but quic-go implementation suffers performance
  • QUIC also designed to multiplex streams over single connection -> not for high performance
  • We want to benefit from other SCION projects (e.g. drkey)

      • Goal: Provide high-performance reliable transport over SCION, implemented in a user-space library to be shared for all kinds of applications

Basic Ideas

  • Part: Block of data that is transferred as a whole
  • Each transferred part may have a different size
  • Parts are transferred over multiple paths (using multiple sockets)
  • If packets are lost, retransfers occur at the end of each part
  • Packets are stored in consecutive memory layout for higher performance

Transfer Process - Overview

Transfer Process - Signaling (per Part)

  • Sender and receiver perform handshake
  • Receiver sends range acks including tuples of (sequenceNumber,count)
  • BBR-like congestion control limits transfer speed per connection
  • In case of packet loss: Sender starts retransferring all received missing packets
  • If all packets of part are transferred, receiver sends partFinished back

Handshake

  • Announce part transfers
  • Exchange host information to improve congestion control
    • Hint the performance of the remote host to detect link congestion or packet dropping on the remote host

Data Transfer

  • Header as small as possible
    • PartId: Identifier of the current part
    • SequenceNumber: Increasing packet number within a part

Announcing Missing Packets

  • Receiver sends ack packet to inform sender about missing packets
    • Often multiple consecutive packets are lost at once
    • Highest sequenceNumber passed back to sender
    • Tx: unique connection identifier

Congestion Control: BBR-like

  • Inspired by BBR for TCP (by Google)
  • Based on state machine:
    • Startup: Increase bandwidth until packet loss occurs
    • Drain: Decrease bandwidth to free full buffers on the path
    • Probe-BW: Increase bandwidth and test for packet loss
    • Probe-RTT: Not used at the moment

Current State: RTS

  • Complete transfer of parts is implemented
  • Congestion control, transfer of partSize < MTU and others need to be optimized
  • And: it's actually not really path-aware
    • Always uses the first available path for all connections - therefor RTS
    • Aim: Use pan library to benefit from all of our path-aware work
    • pan + RTS = PARTS :)
  • But: At least the first performance goals are fulfilled

First benchmark on 10G link

  • Intra-AS transfer, without border router
  • PartSize is 5GB / numConns, so really huge

Future Work

  • Combine with pan to achieve full path-awareness
  • Fine-tune congestion control
  • Implement adaptive part sizing for e.g. data streams
  • Further improve performance
  • Use drkey for optional data encryption

Why do we need multiple sockets?

  • Example: Receiving side, but similar for the sending side
  • Each socket is bound to particular thread
  • If thread is using 100% of the OS thread, reading limits and packets drop
  • And: Reading with multiple threads from one socket hangs in the same limitation
